# 战斗反馈与复盘面板设计

## 设计目标

战斗反馈系统的目标是让玩家看懂三件事：

1. 我生产出来的机器人有没有按预期工作。
2. 我的逻辑规则到底有没有触发、有没有浪费。
3. 这场战斗或这段时间的资源投入是否划算。

头顶气泡负责即时反馈，复盘面板负责统计反馈。两者应该配合，而不是互相替代：

- 头顶气泡告诉玩家“刚才发生了什么”。
- 战斗日志告诉玩家“最近发生了什么”。
- 复盘面板告诉玩家“过去 1/5/10 分钟整体表现如何”。

## 面板入口

建议新增 UI：战斗复盘面板。

MVP 入口放在主界面右下角，与蓝图按钮并列。点击后打开独立统计菜单，左侧显示分类按钮，右侧显示当前分类的统计列表。首版先提供“资源”和“机器人”两个分类，之后再逐步扩展。

后续可以在按钮或底部状态栏旁显示一个简短摘要：

`5 分钟：生产 24 / 损失 9 / 击杀 31 / 净收益 +中`

点击后展开详细列表。

## 时间窗口

你的设想“统计过去指定时间”是正确方向，建议固定为三个窗口：

| 窗口 | 用途 | 设计重点 |
| --- | --- | --- |
| 1 分钟 | 观察当前战斗是否出问题 | 响应快，适合调试刚改的规则 |
| 5 分钟 | 判断一轮进攻或防守是否成功 | 默认窗口，最常用 |
| 10 分钟 | 判断生产、物流、战线整体趋势 | 适合中期基地扩张与巢穴攻坚复盘 |

不建议一开始做任意时间滑条。固定窗口更容易理解，也更容易做性能优化。

## 总览指标

复盘面板顶部显示总览卡片：

| 指标 | 含义 | 玩家能做出的判断 |
| --- | --- | --- |
| 生产机器人数量 | 时间窗口内完成生产并部署的机器人 | 产线是否跟得上损耗 |
| 损失机器人数量 | 时间窗口内死亡、报废或失效的机器人 | 当前战术是否过于消耗 |
| 击杀敌人数量 | 时间窗口内被我方击杀的敌人 | 火力是否有效 |
| 造成伤害 | 我方对敌方造成的总伤害 | 高血量 Boss 战中比击杀更有意义 |
| 承受伤害 | 我方承受的总伤害 | 是否需要护盾、撤退或更重底盘 |
| 资源消耗 | 生产机器人、弹药、维修、投送消耗 | 战斗成本 |
| 战利品收益 | 敌人掉落、残骸回收、巢穴奖励 | 战斗收益 |
| 净收益评价 | 收益减去成本后的粗略判断 | 告诉玩家这场仗是否值得 |

净收益不要只显示精确数字，建议同时显示标签：

- `高收益`：收益明显大于消耗。
- `低收益`：收益略高于消耗。
- `亏损`：消耗高于收益。
- `战略必要`：资源亏损但击败了 Boss、解锁了科技或推进了地图。

这样可以避免玩家因为一次 Boss 战资源亏损而误以为自己“打错了”。

## 机器人生产与损失

机器人统计应该按蓝图快照版本分组，而不是只按当前蓝图名称分组。

原因：

- 蓝图修改后，战场上已有机器人不会改变。
- 产线生产的是蓝图快照。
- 如果只按蓝图名称统计，玩家会无法判断旧版本和新版本谁表现更好。

建议列表字段：

| 字段 | 示例 |
| --- | --- |
| 蓝图名称 | 轻型步枪集结型 |
| 蓝图版本 | v3 |
| 生产数量 | 12 |
| 损失数量 | 5 |
| 击杀数量 | 18 |
| 平均存活时间 | 2 分 14 秒 |
| 资源成本 | 铁板 60、铜线 24、基础电路 12 |
| 回收收益 | 残骸 3、初级掉落物 2 |
| 评价 | 消耗偏高 / 表现稳定 / 适合清杂 |

## 死亡原因统计

死亡原因必须明确，否则玩家很难知道该改生产、改武器、改逻辑还是改物流。

建议初期支持以下死亡或失效原因：

| 原因 | 含义 | 对应改进方向 |
| --- | --- | --- |
| 被击杀 | 生命值归零 | 增加数量、护盾、撤退规则、重底盘 |
| 寿命耗尽 | 临时机器人存在时间结束 | 缩短路程、改部署点、提高生产节奏 |
| 过热宕机 | 热量超过上限导致失效 | 增加停火规则、冷却模块、换武器 |
| 弹药耗尽 | 无法继续攻击并被判定失效或撤退失败 | 建立弹药补给、前线补给点、物流规则 |
| 能量耗尽 | 电池耗尽导致停机 | 增加电池、能源补给、降低高耗能动作 |
| 迷路/卡死 | 长时间无法到达目标 | 检查地图通路、集结点、寻路规则 |
| 无目标闲置 | 长时间没有有效目标或动作 | 检查条件优先级、目标筛选、侦察范围 |

其中“弹药耗尽”“过热宕机”不一定都必须直接死亡，也可以是战斗失效。复盘面板中可以统一放入“损失与失效原因”，比单纯叫“死亡原因”更准确。

## 规则触发统计

规则触发统计是逻辑玩法的核心调试工具。

这里的“触发”表示机器人从其它状态进入一条新的匹配规则，而不是规则条件持续为真时每次轮询都重复记账。例如机器人连续等待队友 10 秒，应记录为一次“等待队友”触发，并额外统计执行时长，而不是按帧数或 Tick 数累计。

每条规则记录：

| 字段 | 含义 |
| --- | --- |
| 规则名称 | 玩家自定义或系统生成的规则名 |
| 所属蓝图版本 | 哪个蓝图快照上的规则 |
| 触发次数 | 时间窗口内成功触发次数 |
| 成功执行次数 | 触发后动作成功开始的次数 |
| 中断次数 | 因死亡、目标丢失、路径失败等中断 |
| 平均执行时长 | 动作持续多久 |
| 触发后结果 | 触发后短时间内的击杀、撤退成功、存活变化 |
| 标记变化 | 规则是否设置或清除了战术标记 |

面板中建议单独标记：

- `高频触发`：触发次数特别多，可能是核心规则，也可能是死循环。
- `从未触发`：规则可能条件太苛刻或顺序太低。
- `触发但失败`：条件成立，但动作无法完成。
- `被更高优先级覆盖`：规则本来可触发，但被前面的规则抢先执行。
- `状态卡住`：规则设置了某个标记，但后续规则没有接管，机器人长期停留在同一状态。

这比单纯显示“触发了多少次”更有用。

## 战术标记诊断

战术标记是 If-Then 规则表达顺序行为的关键，因此复盘面板需要能解释标记变化，否则玩家很难判断“先集结，再战斗”为什么没发生。

建议记录：

| 字段 | 含义 |
| --- | --- |
| 机器人 ID | 哪个单位发生了标记变化 |
| 蓝图版本 | 哪个蓝图快照上的规则 |
| 标记名称 | 例如 `已集结`、`正在撤退` |
| 旧值 / 新值 | 从 `否` 到 `是`，或从 `是` 到 `否` |
| 触发规则 | 哪条规则导致变化 |
| 持续时间 | 该标记维持了多久 |

常见诊断：

- `从未置位`：到达条件太苛刻，机器人一直无法完成集结。
- `反复置位/清除`：规则互相打架，可能需要更高优先级或冷却。
- `置位后无后续动作`：缺少“已集结后默认战斗/推进”的规则。

## 敌人造成损失统计

敌人统计用于回答“到底谁在杀我的机器人”。

建议字段：

| 字段 | 示例 |
| --- | --- |
| 敌人类型 | 重甲犀牛 |
| 击杀我方数量 | 7 |
| 造成伤害 | 3200 |
| 主要受害蓝图 | 轻型步枪集结型 v3 |
| 我方击杀该敌人数 | 3 |
| 推荐排查方向 | 需要穿甲/激光/更重底盘 |

这一栏可以和科技树、敌人图鉴联动：

- 如果重甲敌人造成大量损失，提示查看装甲克制。
- 如果空中敌人造成大量损失，提示检查防空武器或目标优先级。
- 如果干扰敌人造成大量失效，提示检查广播、标记、传感器升级。

## 资源消耗与收益

资源统计不要只看“击杀收益是否大于生产成本”，因为某些战斗的目标是阶段推进。

建议分成三层：

| 层级 | 内容 | 用途 |
| --- | --- | --- |
| 直接成本 | 机器人生产材料、弹药、能量、维修、投送物资 | 判断战斗亏不亏 |
| 直接收益 | 敌人掉落物、残骸回收、巢穴资源、地图资源点 | 判断是否值得刷 |
| 战略收益 | 解锁科技、清除威胁圈、打开新资源区、击败 Boss | 判断是否值得一次性投入 |

显示示例：

| 项目 | 数量 |
| --- | --- |
| 消耗 | 铁板 180、铜线 70、动能弹药箱 12 |
| 回收 | 残骸碎片 9、初级感应线圈 1 |
| 击杀收益 | 普通掉落物 14 |
| 战略收益 | 清除小型巢穴、解锁阶段 1 科技 |
| 评价 | 战术亏损，战略收益高 |

这种表达能保护 Boss 战和首次攻坚的体验：玩家可以接受资源亏损，只要系统明确告诉他“这不是刷资源行为，而是推进行为”。

## UI 列表结构

完整复盘面板建议采用左侧分类菜单，而不是所有数据堆在一张表里。

| 标签页 | 内容 |
| --- | --- |
| 总览 | 生产、损失、击杀、成本、收益 |
| 蓝图 | 按蓝图版本统计表现 |
| 规则 | 规则触发、未触发、失败、中断 |
| 死因 | 被击杀、寿命耗尽、过热、弹药耗尽等 |
| 敌人 | 哪种敌人造成最多损失 |
| 资源 | 成本、掉落、回收、战略收益 |

默认打开“总览”，并在每个标签页顶部保留时间窗口切换：

`1 分钟 | 5 分钟 | 10 分钟`

## 即时反馈规则

头顶气泡仍然保留，但应该克制显示频率。

建议只显示这些事件：

- 关键规则首次触发。
- 机器人开始撤退。
- 机器人进入集结状态。
- 机器人弹药不足。
- 机器人热量过高并停火。
- 机器人收到广播指令。
- 机器人锁定 Boss 或高价值目标。

同一机器人同一规则不要连续刷屏，可以设置 3-5 秒冷却。

气泡文本应短：

- `集结`
- `撤退`
- `弹药不足`
- `过热停火`
- `收到广播`
- `锁定目标`

详细原因交给战斗日志和复盘面板。

## 数据记录建议

实现上建议采用事件流，而不是每帧扫描全局状态。

每当关键事件发生时写入一条轻量事件：

| 事件类型 | 必要字段 |
| --- | --- |
| robot_produced | 时间、队伍、机器人 ID、蓝图 ID、蓝图版本、资源成本 |
| robot_lost | 时间、机器人 ID、蓝图版本、原因、击杀者敌人类型 |
| enemy_killed | 时间、敌人类型、击杀者机器人 ID、掉落物 |
| rule_triggered | 进入新规则的时间、机器人 ID、蓝图版本、规则 ID、规则名称 |
| rule_failed | 时间、机器人 ID、规则 ID、失败原因 |
| state_flag_changed | 时间、机器人 ID、蓝图版本、标记名称、旧值、新值、规则 ID |
| resource_spent | 时间、资源类型、数量、用途 |
| resource_gained | 时间、资源类型、数量、来源 |
| damage_dealt | 时间、来源 ID、目标类型、伤害量、伤害类型 |
| damage_taken | 时间、机器人 ID、敌人类型、伤害量、伤害类型 |

面板查询时只统计当前窗口内的事件。

为了避免长期存档膨胀：

- 只保留最近 10 分钟的详细事件。
- 重要战斗结束时可以生成一份摘要报告。
- Boss 战、巢穴清除、基地被袭击可以单独保存关键摘要。

## 与科技树的关系

复盘功能也可以分阶段解锁：

| 科技 | 解锁内容 |
| --- | --- |
| 基础战斗报告 | 总览、生产、损失、击杀 |
| 规则诊断 | 规则触发次数、未触发规则 |
| 战术分析终端 | 敌人造成损失、伤害类型统计 |
| 资源审计模块 | 成本、收益、战略收益评价 |
| 高级战场记录器 | 按蓝图版本、规则结果、Boss 战摘要分析 |

这样早期不会给玩家过多数据，中后期又能支撑复杂自动化调试。

## 设计判断

你的统计面板方案是合适的，而且应该成为逻辑玩法的核心支撑系统。

需要注意的是，面板不应该只是“数据仓库”，而应该回答具体问题：

- 如果机器人死得多，告诉玩家主要死因。
- 如果规则没用，告诉玩家它从未触发、触发失败，还是被更高优先级覆盖。
- 如果战斗亏损，告诉玩家是战术亏损还是战略投入。
- 如果某种敌人造成大量损失，引导玩家去看武器克制、底盘升级或逻辑规则。

复盘面板的最终目标不是让玩家看更多数字，而是让玩家敢于修改蓝图和规则，并能确信修改确实改变了战斗结果。
