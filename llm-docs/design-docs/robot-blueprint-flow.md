# Logic Line 机器人蓝图生命周期设计

本文档定义“机器人蓝图”作为生产、UI、存档、部署和战斗之间的核心数据对象。蓝图不是单台机器人，而是玩家设计的一种可重复生产的机器人型号。

核心原则：

`蓝图是设计；产线复制蓝图快照；机器人是一次性实例。`

---

## 一、机器人蓝图包含什么

一个完整机器人蓝图应包含以下信息：

1. **蓝图元数据**
   * 蓝图名称。
   * 蓝图图标或颜色。
   * 蓝图类型：战斗机器人 / 物流机器人 / 混合特种机器人。
   * 蓝图版本号。
   * 创建时间与最后修改时间。

2. **底盘配置**
   * 底盘 ID。
   * 底盘类型：战斗底盘 / 物流底盘。
   * 基础属性：HP、速度、寿命、护甲、可用插槽。
   * 底盘提供的被动能力，例如免疫击退、飞越地形、基础护盾。

3. **模块配置**
   * 武器槽：主武器、副武器、主动技能武器。
   * 防御槽：护盾、装甲、诱饵。
   * 工具槽：维修焊枪、目标锁定器、回收扫描器。
   * 背包槽：物流容量模块、压缩货舱。
   * 特殊槽：远程投送模块、残骸牵引器。

4. **逻辑配置**
   * 逻辑板 ID。
   * 可用自定义规则槽数量。
   * 规则文件路径或内嵌规则数组。
   * 默认脑干是否启用。
   * 规则版本号。

5. **生产配方**
   * 需要哪些底盘、武器、模块、逻辑板。
   * 生产耗时。
   * 生产建筑要求。
   * 是否需要特殊掉落物。

6. **部署策略**
   * 出厂后部署位置：主基地、前线补给点、指定集结点。
   * 默认任务：进攻、防守、物流、回收、补给。
   * 允许部署区域。
   * 是否自动补位。

## 二、蓝图能否保存、复制、升级

### 1. 蓝图可以保存

玩家可以保存任意数量的机器人蓝图。游戏不应在蓝图数量上强行卡玩家，因为蓝图本质上是玩家的策略笔记和生产模板。

保存内容包括：

- 装配配置。
- 逻辑规则。
- 部署策略。
- 生产配方引用。

### 2. 蓝图可以复制

复制蓝图是鼓励玩家实验的关键能力。

典型使用场景：

- 从“基础步枪兵”复制出“集结步枪兵”。
- 从“战地搬运车”复制出“高危回收车”。
- 从“导弹车”复制出“防干扰导弹车”。

复制后，新蓝图拥有独立名称、独立规则和独立版本。

### 3. 蓝图可以升级

升级分两种：

#### 直接编辑升级

玩家打开蓝图，替换底盘、武器、模块或规则。

适合：

- 小规模修改。
- 修正错误规则。
- 替换更高级模块。

#### 派生升级

玩家以旧蓝图为基础创建新版本。

适合：

- 保留旧产线。
- A/B 测试不同逻辑。
- 避免修改导致整个前线生产行为变化。

推荐 UI 提供两个按钮：

- `保存修改`
- `另存为新蓝图`

## 三、蓝图修改后，战场机器人是否改变

**不改变。**

已经在战场上的机器人是由生产线生成的临时实例。它们拥有出生时复制下来的蓝图快照，不会因为玩家修改蓝图而热更新。

理由：

- 机器人寿命有限，热更新意义不大。
- 避免玩家通过暂停修改蓝图来远程微操战场单位。
- 让战场结果更可读：某一波机器人失败，是因为那一波生产时使用了旧设计。
- 保留自动化游戏的“迭代感”：修改蓝图后，要等下一批新机器人生产出来才会看到效果。

### 战场实例应保存的快照信息

每台机器人生成时，应复制：

- 底盘属性。
- 模块属性。
- 逻辑规则数组。
- 部署目标。
- 生产批次 ID。
- 蓝图版本号。

战斗报告中可以显示：

`蓝图：集结步枪兵 v3`

这样玩家能知道是哪一版设计出了问题。

## 四、产线生产的是快照还是实时引用

**产线生产当前蓝图快照。**

这意味着当玩家把某个蓝图绑定到机器人锻造厂时，该锻造厂会记录一份当时的蓝图快照。之后玩家修改蓝图，不会自动影响已绑定产线，除非玩家手动点击“更新产线蓝图”。

### 为什么不使用实时引用

实时引用会带来几个问题：

- 玩家改蓝图时，所有产线同时变化，容易造成大规模意外。
- 很难做 A/B 测试。
- 难以解释为什么某条产线突然生产出不同单位。
- 存档和战斗复盘会变复杂。

### 推荐产线状态

每条机器人锻造厂产线应显示：

- 当前绑定蓝图名称。
- 蓝图版本。
- 是否落后于最新版本。
- 当前生产队列。
- 生产该蓝图需要的材料是否充足。

当蓝图被修改后，产线 UI 显示：

`当前产线使用：集结步枪兵 v2`

`蓝图库最新：集结步枪兵 v3`

玩家可以选择：

- 继续生产旧版本。
- 更新到最新版本。
- 复制产线并绑定新版本。

## 五、逻辑插槽如何限制规则数量

你的回答是：每一个逻辑插槽增加规则数量。这个方向可行，但需要加上限制，否则规则数量会变成“越多越强”的线性数值，玩家最终可能被迫维护一大坨规则。

推荐设计：

**逻辑插槽限制的是玩家自定义规则数量，而不是默认脑干。**

### 1. 默认脑干永久存在

所有作战机器人都有不可删除的默认脑干：

`寻找目标 -> 接近目标 -> 开火`

所有基础物流机器人也有不可删除的默认脑干：

`寻找任务 -> 搬运物品 -> 返回仓库`

默认脑干不占用逻辑插槽。

玩家自定义规则用于“插队”和“覆盖默认行为”。

### 2. 逻辑插槽提供自定义规则槽

示例：

| 逻辑板 | 自定义规则槽 | 典型用途 |
|---|---:|---|
| 无逻辑板 | 0 | 只运行默认脑干 |
| 基础逻辑控制板 | 1 | 集结、低血撤退、优先目标之一 |
| 中级逻辑控制板 | 3 | 热量控制、弹药判断、目标筛选 |
| 高级战略逻辑板 | 5 | 协同、广播、特种反制 |
| 终极 AI 矩阵 | 不限或 10+ | 终局 Combo 与复杂战术 |

### 2.1 战术标记槽

If-Then 规则需要少量“记忆”，用于表达顺序行为，例如“先到集结点，再允许默认脑干战斗”。因此蓝图还应记录可用的战术标记槽。

第一版建议只开放自身布尔标记：

- 基础逻辑控制板：1 个自身标记槽，默认可命名为 `已集结`。
- 中级逻辑板：2-3 个自身标记槽，用于 `已集结`、`正在撤退`、`已接敌` 等简单状态。
- 高级战略逻辑板：可以解锁目标标记、广播标记或小队共享标记。

标记不等于任意变量。MVP 阶段不提供数字变量、字符串变量、循环或脚本，只允许规则读取和设置有限布尔标记。

典型规则：

`If 自身标记[已集结] = 否 & 距离集结点 > 20 -> Then 前往集结点`

`If 自身标记[已集结] = 否 & 距离集结点 <= 20 -> Then 设置自身标记[已集结] = 是`

`If 自身标记[已集结] = 是 -> Then 默认脑干接管`

战术标记属于机器人运行时状态。蓝图保存的是“有哪些标记槽、标记名称、哪些规则读写它们”；每台从蓝图快照生产出来的机器人拥有自己的标记值，出生时按蓝图默认值初始化。

### 3. 为什么不能简单堆很多规则

规则越多确实越聪明，但也会产生问题：

- 玩家难以判断哪条规则正在生效。
- 高优先级规则可能永远挡住低优先级规则。
- 多条规则之间容易互相打架。
- UI 会变长，调试成本上升。
- 战斗失败后很难定位问题。

所以规则槽应当是一种“表达能力限制”，不是越多越好的纯数值。

### 4. 推荐引入规则复杂度

长期可以让逻辑板限制的不只是规则条数，还包括“复杂度预算”。

例如：

- 简单条件：1 点复杂度。
- 标签判断：1 点复杂度。
- 周围友军数量：2 点复杂度。
- 广播/接收信号：3 点复杂度。
- 多动作连招：额外消耗复杂度。

这样可以避免玩家在基础逻辑板上塞入一条过于复杂的超级规则。

第一版可以先不实现复杂度预算，只保留文档概念。

### 5. 推荐最小实现

第一版规则限制：

- 基础逻辑板：1 条自定义规则。
- 中级逻辑板：3 条自定义规则。
- 高级逻辑板：5 条自定义规则。
- 基础逻辑板至少提供 1 个自身布尔标记槽，用于 `已集结` 这类顺序行为。
- 默认脑干不占规则槽。
- 规则从上到下判定，第一条满足后执行。
- UI 显示“规则槽 1/3”。

这个方案足够清晰，也便于玩家理解规则数量如何影响机器人表现。

## 六、蓝图与规则文件的关系

当前项目已经存在 JSON 规则导出机制。未来蓝图可以有两种保存方式：

### 1. 蓝图引用规则文件

蓝图中保存：

`rules_path = "res://debug_exports/rules/logic_rules_001.json"`

优点：

- 便于调试。
- 规则可以独立导入导出。
- 适合开发期。

缺点：

- 玩家层面的蓝图存档可能变得分散。
- 修改规则文件可能影响多个蓝图引用。

### 2. 蓝图内嵌规则数据

蓝图直接保存规则数组。

优点：

- 蓝图是完整对象。
- 存档更稳定。
- 复制蓝图时规则自然复制。

缺点：

- 调试时不如独立 JSON 直观。

### 推荐方案

开发期：

- 允许蓝图引用外部 JSON。
- 用于快速调试规则。

正式玩家存档：

- 蓝图内嵌规则数据。
- 导出时可以额外生成 JSON，供高级玩家分享或调试。

## 七、蓝图升级与科技树

蓝图升级不应该只是把数值变大，而应当拓宽可能性。

### 1. 科技解锁如何影响蓝图

科技可以解锁：

- 新底盘。
- 新模块。
- 新逻辑条件。
- 新逻辑动作。
- 更多规则槽。
- 新部署策略。

### 2. 旧蓝图如何升级

当科技解锁后，旧蓝图不会自动变化。玩家可以：

- 打开旧蓝图手动替换模块。
- 使用“升级建议”自动生成派生蓝图。
- 继续保留低成本旧蓝图用于炮灰或后勤。

### 3. 示例

旧蓝图：

`轻型轮式底盘 + 轻型步枪 + 基础逻辑控制板`

升级路线：

- 替换为中级逻辑板，增加低血撤退规则。
- 替换轻型步枪为热熔激光，增加过热停火规则。
- 派生出“集结步枪兵”和“自保步枪兵”两个不同版本。

## 八、UI 与玩家体验要求

蓝图 UI 是核心玩法入口，不应藏在调试面板里。正式游戏界面右下角应提供一个常驻“蓝图”按钮，玩家点击后进入机器人自定义蓝图管理界面。

### 1. 蓝图库

玩家应有一个蓝图库，显示所有保存的蓝图：

- 名称。
- 类型。
- 使用底盘。
- 主要武器或功能模块。
- 逻辑槽使用情况。
- 绑定到哪些产线。
- 最新版本号。

蓝图库基础操作：

- 新建蓝图。
- 编辑选中蓝图。
- 复制蓝图。
- 标记收藏或废弃。
- 查看哪些机器人锻造厂正在使用该蓝图。

### 2. 蓝图编辑器

蓝图编辑器至少包含：

- 底盘区。
- 模块槽位区。
- 逻辑规则区。
- 生产成本预览。
- 部署策略区。
- 保存 / 另存为 / 更新产线按钮。

推荐编辑流程：

1. **选择底盘**：先决定机器人定位，例如轻型战斗底盘、物流底盘或重型底盘。
2. **选择功能模块**：根据底盘槽位安装武器、护盾、背包、工具或特殊模块。
3. **配置 If-Then 规则**：设置自定义规则、战术标记、默认脑干是否启用。
4. **命名并保存**：输入蓝图名称，保存为新蓝图或覆盖当前蓝图版本。

这个顺序很重要：底盘决定槽位，模块决定可用动作，逻辑板决定规则数量和可用条件。如果玩家先写规则，后选模块，会出现规则引用不存在动作的问题。

MVP 阶段可以做成单页编辑器，但视觉顺序仍应保持：

`底盘与模块 -> If-Then 规则 -> 名称与保存`

### 3. 产线绑定面板

机器人锻造厂需要清晰显示：

- 当前生产的蓝图快照。
- 是否有新版本可用。
- 每分钟预计产量。
- 缺少哪些材料。
- 生产出的机器人部署到哪里。

锻造厂面板需要提供“选择蓝图”按钮。点击后打开二级弹出菜单或列表：

- 显示所有可生产蓝图。
- 显示蓝图名称、版本、主要底盘和成本摘要。
- 蓝图很多时允许滚动、搜索或按类型筛选。
- 选择后锻造厂绑定该蓝图的当前快照。

绑定后，锻造厂不直接引用正在编辑的蓝图草稿。玩家后续修改蓝图时，锻造厂继续生产旧快照，直到玩家手动更新或重新选择蓝图。

### 4. 战斗复盘

战斗报告应按蓝图版本统计：

- 生产数量。
- 存活时间。
- 击杀数。
- 死亡原因。
- 规则触发次数。
- 是否有规则从未触发。

这能帮助玩家判断是产线问题、配方问题、逻辑问题还是部署问题。

## 九、风险与建议

### 风险 1：蓝图无限保存导致 UI 混乱

你不希望在数量上卡玩家，这是好的。但 UI 必须提供管理能力：

- 搜索。
- 标签。
- 收藏。
- 按底盘/用途筛选。
- 标记废弃蓝图。

否则后期蓝图会变成文件垃圾堆。

### 风险 2：快照机制让玩家困惑

玩家可能会问：我明明改了蓝图，为什么产线还在生产旧版本？

解决方法：

- 产线 UI 明确显示版本差异。
- 蓝图保存后弹出提示：“已有产线不会自动更新。”
- 提供一键“将所有使用旧版本的产线更新到 v3”。

### 风险 3：规则数量越多越强

如果每个逻辑插槽只增加规则数量，玩家可能被迫堆规则。

解决方法：

- 默认脑干负责基础行为。
- 规则槽只用于关键覆盖。
- 高阶逻辑板解锁新类型，而不只是加数量。
- 战斗报告显示规则触发效果，帮助玩家删除无效规则。

### 风险 4：蓝图升级变成纯数值替换

如果升级只是“轻型底盘 -> 重型底盘”，玩家不会产生设计感。

解决方法：

- 新底盘改变槽位结构。
- 新逻辑板解锁新条件或动作。
- 新武器带来新的生产与后勤约束。
- 新部署策略改变机器人进入战场的方式。

---

## 十、最终建议

采用以下规则作为蓝图生命周期的基准：

1. 蓝图可以无限保存、复制、升级。
2. 已在战场上的机器人不受蓝图修改影响。
3. 产线生产蓝图快照，不使用实时引用。
4. 蓝图修改后，产线需要手动更新快照。
5. 逻辑插槽限制自定义规则数量，默认脑干不占槽。
6. 高级逻辑板不只增加规则数量，还应解锁新条件、新动作和新部署策略。
7. 战斗复盘必须按蓝图版本统计表现。

这样可以让玩家放心实验，又不会让生产线和战场行为因为蓝图修改而变得不可预测。
