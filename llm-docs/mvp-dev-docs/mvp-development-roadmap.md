# MVP 分段开发任务与素材需求

本文档用于指导 First Playable MVP 的实际开发。它不是新的玩法设计，而是把现有设计拆成可执行工程阶段。

目标玩家体验来自 [first-playable-mvp.md](first-playable-mvp.md)：

`生产机器人 -> 默认 AI 失败 -> 修改逻辑规则 -> 集结后突破敌巢`

开发目标：

- 每个阶段都能运行、能验证、能回退。
- 代码尽可能模块化，避免把经济、战斗、AI、UI 写进同一个 `GameManager`。
- UI 先服务于可读性和调试效率，不追求最终品质。
- 美术素材先服务于可读性，不追求最终品质。
- 每个阶段结束时都留下可自动化验证的入口。

## 一、总体开发策略

MVP 推荐按“先骨架、再系统、后表现”的顺序开发：

1. 工程地基：目录、数据结构、事件系统、测试入口。
2. 视口地基：`GameWorld`、`MainCamera`、`CanvasLayer UI` 三层分离。
3. 地图地基：类似 `yorg.io` 的网格地图，建筑按格占用，机器人自由移动。
4. 经济闭环：主基地库存、采矿、加工、抽象物流。
5. 生产闭环：机器人锻造厂、蓝图绑定、自动补位。
6. 战斗闭环：机器人、敌人、武器、死亡、寿命。
7. 规则闭环：默认脑干、自定义规则、集结点。
8. 关卡闭环：敌巢、守军、胜利条件。
9. 反馈闭环：事件日志、复盘摘要、调试 UI。

不要一开始做完整建筑摆放、完整物流、完整科技树或漂亮 UI。MVP 的核心是验证“规则修改能改变战斗结果”。

UI 不是最后包装，而是开发工具本身。每个系统阶段都必须同步交付一个最小 UI 面板，让开发者能看见状态、等待原因、事件和关键数值。

## 二、MVP UI 开发原则

MVP UI 的目标不是好看，而是减少调试痛苦：

- 信息层级清楚。
- 状态一眼可读。
- 操作路径短。
- 每个核心系统都有可观察面板。
- 调试 UI 后续能自然演化成正式 UI。

### 1. 推荐屏幕布局

第一版不做复杂窗口系统，采用稳定的开发期三栏布局：

```text
顶部：资源条 + 游戏时间 + 当前目标

左侧：选中对象检查器
中间：游戏地图
右侧：调试事件与复盘面板

底部：当前选中对象的快捷操作
```

布局职责：

- 顶部永远显示资源和目标。
- 左侧显示当前选中建筑、机器人或敌巢状态。
- 右侧常驻事件、复盘、规则触发、生产状态。
- 底部只放当前对象的关键操作，例如设置集结点、调整目标存活数量。

### 2. 视觉风格

先做“专业、安静、清楚”：

- 深色或低饱和背景。
- 浅色文字。
- 蓝色表示玩家操作或选中。
- 黄色表示等待或警告。
- 红色表示错误、受损、资源不足。
- 绿色表示运行中、完成、收益。
- 面板圆角控制在 8px 以内。
- 表格行高、按钮高度、内边距保持一致。
- 图标不足时先用文字，不使用难懂的临时图。

### 3. UI 与系统解耦

- UI 读取 `Inventory`、`RobotForge`、`CombatEventLog` 的公开数据。
- UI 通过信号或命令调用系统，例如 `set_rally_point(position)`。
- UI 不直接修改库存数字、机器人 HP 或事件列表。
- 复盘统计函数与面板显示分离，便于 headless 测试。

## 三、MVP 地图开发原则

MVP 地图参考 `yorg.io` 的表达方式：

- 整张地图由规则网格构成。
- 建筑以图标形式显示，并占用网格。
- 普通建筑默认占用 `1x1` 格。
- 特殊建筑可以占用更大矩形，例如 `2x2`、`3x3`。
- 资源点占用网格，但可以显示为矿石图标。
- 机器人、敌人、子弹不受网格限制，使用连续世界坐标自由移动。
- UI 与摄像机不属于地图网格。

### 1. 双坐标系统

地图必须同时支持两套坐标：

| 坐标 | 用途 | 示例 |
| --- | --- | --- |
| 网格坐标 `Vector2i` | 建筑放置、资源点、占格检测 | 主基地在 `(20, 18)` |
| 世界坐标 `Vector2` | 机器人移动、子弹、战斗距离、摄像机 | 机器人位置 `(1320.5, 910.0)` |

推荐提供统一转换函数：

```text
grid_to_world(cell: Vector2i) -> Vector2
world_to_grid(position: Vector2) -> Vector2i
get_cell_rect(origin: Vector2i, size: Vector2i) -> Rect2
```

建筑保存网格坐标和占格尺寸，机器人只保存世界坐标。机器人可以把建筑中心点、集结点、敌巢中心点作为移动目标。

### 2. 建筑占格规则

MVP 建筑占格先使用矩形：

| 对象 | 默认占格 |
| --- | --- |
| 主基地 | `2x2` |
| 采矿机 | `1x1` |
| 基础加工厂 | `1x1` |
| 机器人锻造厂 | `1x1` |
| 前线补给点，MVP 后 | `1x1` 或 `2x1` |
| 敌巢 | `2x2` |

占格系统必须能回答：

- 某个格子是否为空。
- 某个矩形区域是否可以放置建筑。
- 某个世界坐标对应哪个格子。
- 点击某个格子时选中了建筑、资源点，还是空地。

### 3. 地图层级

`GameWorld` 内建议固定拆层：

```text
GameWorld
├── TerrainLayer       # 网格背景、地块颜色
├── ResourceLayer      # 铁矿、铜矿等资源点
├── BuildingLayer      # 主基地、采矿机、加工厂、锻造厂
├── RallyLayer         # 集结点、范围提示、选择框
├── EnemyLayer         # 敌巢、敌方单位
├── UnitLayer          # 玩家机器人
└── ProjectileLayer    # 子弹、命中特效
```

这样做的目的：

- 地图格子和建筑选择不会和机器人移动混在一起。
- 子弹、机器人、敌人可以继续自由移动。
- 后续添加小地图、战争迷雾、地图分区时更容易。

### 4. MVP 地图尺寸

第一版不需要真正巨大，但要比屏幕大，让摄像机和坐标系统从一开始就按“大地图”思路工作。

建议：

| 项目 | MVP 值 |
| --- | --- |
| 格子尺寸 | `64x64` 像素 |
| 测试地图尺寸 | `64x64` 格 |
| 世界尺寸 | `4096x4096` 像素 |
| 初始摄像机视口 | 看向基地附近 |
| 初始资源点 | 基地附近 1 个铁矿、1 个铜矿 |
| 第一敌巢 | 距基地约 12-18 格 |

阶段 0 可以先画完整网格背景；阶段 2-3 再实现点击、选择、占格和资源点。

### 5. 地图交互优先级

当玩家点击地图时，推荐优先级：

1. UI 面板拦截点击。
2. 地图对象选择：建筑、资源点、敌巢、机器人。
3. 当前处于设置集结点模式时，将点击位置转换为集结点。
4. 当前处于建造模式时，尝试在该格放置建筑。
5. 否则选中空格，显示格子坐标。

MVP 不需要完整建造菜单，但应尽早建立这套点击流。

## 五、模块化原则

### 1. `GameManager` 只做编排

`GameManager` 不应该直接保存所有经济、战斗、AI 细节。

允许它负责：

- 加载测试关卡。
- 持有主要系统引用。
- 创建初始建筑和敌巢。
- 判断 MVP 胜利条件。

不建议它负责：

- 资源扣减细节。
- 机器人 AI 判断。
- 子弹伤害计算。
- 事件统计。
- UI 表格组装。

### 2. 数据用 Resource 或 Dictionary，逻辑用节点/服务

适合做成数据：

- 资源定义。
- 配方。
- 机器人蓝图。
- 机器人属性。
- 建筑定义。
- 敌人属性。

适合做成逻辑模块：

- 库存增减。
- 生产队列。
- AI 规则解析。
- 伤害结算。
- 事件统计。

### 3. 组件优先于继承层级

机器人和敌人都需要 HP、移动、攻击、队伍、死亡事件。建议尽量抽成组件：

- `HealthComponent`：HP、受伤、死亡信号。
- `MovementComponent`：移动到位置、停止。
- `VisibleEnemySensor`：从当前可见地图范围扫描敌军列表，不使用雷达碰撞箱。
- `WeaponComponent`：冷却、射击、生成子弹。
- `LifespanComponent`：寿命倒计时、寿命耗尽事件。

这样后面物流机器人、重型机器人、敌人单位都能复用。

### 4. 系统之间通过事件解耦

建议新增事件流：

- `robot_produced`
- `robot_lost`
- `enemy_killed`
- `rule_triggered`
- `rule_failed`
- `resource_spent`
- `resource_gained`
- `nest_destroyed`

战斗报告只监听事件，不直接查询所有机器人和敌人。

### 5. UI 面板按职责拆分

推荐 MVP UI 面板：

| 面板 | 职责 | 何时出现 |
| --- | --- | --- |
| `Hud` | 顶部资源、时间、当前目标 | 常驻 |
| `ResourcePanel` | 资源数量、每分钟变化量、缺口提示 | 常驻或顶部 |
| `ObjectInspector` | 当前选中对象的通用信息 | 左侧 |
| `BuildingStatusPanel` | 建筑状态、配方、进度、等待原因 | 选中建筑 |
| `ForgePanel` | 蓝图、目标存活数量、生产进度、集结点 | 选中锻造厂 |
| `RobotInspector` | HP、寿命、目标、当前规则、默认脑干状态 | 选中机器人 |
| `DebugEventPanel` | 最近事件、事件过滤、payload 展开 | 右侧常驻 |
| `CombatReportPanel` | 5 分钟摘要、死亡原因、规则统计 | 右侧或胜利后 |

## 六、阶段 0：工程整理与基线验证

### 目标

在正式扩展前，确认现有战斗 MVP、逻辑 UI、JSON 规则加载仍然可运行。

### 功能任务

- 梳理现有 `Scripts/`、`Scenes/`、`Resources/`。
- 保留现有 `robot.gd`、`ai_controller.gd`、`ai_rule.gd`、`ai_condition.gd` 的行为。
- 确认当前逻辑规则 UI 可以导出 JSON。
- 确认机器人可以加载 `logic_rules_001.json` 和 `logic_rules_002.json`。
- 新增或整理一个 MVP 测试主场景，例如 `Scenes/mvp/mvp_test_map.tscn`。
- 建立基础场景层级：
  - `GameWorld`：承载地图、建筑、机器人、敌人、子弹等世界对象。
  - `MainCamera`：维护当前游戏画面。
  - `MvpHUD`：使用 `CanvasLayer`，不随摄像机移动。
- 第一版 `MainCamera` 固定在 1920x1080 测试地图中心。
- 建立网格地图骨架：
  - `GridMap` 或同等职责节点负责地图尺寸、格子尺寸、坐标转换。
  - `TerrainLayer` 先绘制深色网格背景。
  - `BuildingLayer`、`ResourceLayer`、`UnitLayer` 等层级先建空节点。
  - 第一版地图尺寸使用 `64x64` 格，每格 `64x64` 像素。
- 地图背景不再只是屏幕背景，而是 `4096x4096` 的世界区域。

### UI 交付

- 建立 MVP HUD 根节点，例如 `Scenes/ui/mvp_hud.tscn`。
- 放置顶部资源条占位、左侧对象检查器占位、右侧调试面板占位。
- 保证 UI 不遮挡地图核心区域。
- 确认摄像机移动或后续缩放时，HUD 仍停留在屏幕空间。
- 现有逻辑规则 UI 继续可用，并保留中文标签。

推荐文件：

```text
Scenes/ui/mvp_hud.tscn
Scripts/ui/hud.gd
Scripts/ui/object_inspector.gd
Scripts/ui/debug_event_panel.gd
Scenes/mvp/mvp_test_map.tscn
Scenes/map/grid_map.tscn
Scripts/map/grid_map.gd
Scripts/map/grid_overlay.gd
```

### 模块化要求

- 不在这个阶段重写大系统。
- 只建立新目录约定和最小兼容层。
- 世界对象必须逐步放入 `GameWorld`，不要直接散落在根节点下。
- UI 必须放在 `CanvasLayer` 中，不能作为 `GameWorld` 子节点。
- `MainCamera` 只负责世界视图，不参与 UI 布局。
- 地图网格只约束建筑和资源点，不约束机器人、敌人、子弹移动。
- 所有建筑类对象必须保存 `grid_origin` 和 `grid_size`，即使阶段 0 暂时没有真实建筑。
- 任何迁移都要保持现有 smoke test 能运行。

### 美术素材需求

阶段 0 不需要新美术。可以继续使用现有占位素材：

- 机器人临时图标。
- 子弹临时纯色形状。
- UI 默认控件。
- UI 使用统一深色面板和默认字体即可。
- 背景使用 `64x64` 格的深色网格，占位覆盖 `4096x4096` 世界区域。

### 验收标准

- 主场景可运行。
- 逻辑 UI 可打开。
- 规则 JSON 可导出。
- 机器人仍能按已有规则行动。
- MVP HUD 占位可以显示，不遮挡地图核心区域。
- `MainCamera` 可以看到测试地图的一部分，而不是整个世界。
- 深色背景与摄像机视口对齐，不露出默认灰底。
- 网格线随摄像机移动属于世界内容，HUD 不随摄像机移动。
- 可以通过函数完成世界坐标与网格坐标互转。

## 七、阶段 1：核心数据与事件地基

### 目标

建立后续所有系统共用的数据模型和事件接口。这个阶段看起来“不像游戏”，但能显著减少后面返工。

### 功能任务

- 新增 `CombatEventLog`。
- 新增 `DebugEventPanel`：
  - 显示最近 20 条事件。
  - 支持按事件类型过滤。
  - 点击事件可展开 payload。
  - 支持切换 1/5/10 分钟窗口，MVP 可先只实现 5 分钟统计。
- 新增统一 ID 分配方式，例如 `IdProvider.next_id(prefix)`。
- 定义资源 ID：
  - `iron_ore`
  - `copper_ore`
  - `iron_plate`
  - `copper_wire`
  - `construction_mass`
- 定义配方数据结构：
  - 输入资源。
  - 输出资源或单位。
  - 耗时。
  - 配方类型：资源加工 / 建筑建造 / 单位生产。
  - 目标 ID，例如建筑 ID 或机器人蓝图 ID。
- 将所有 MVP 配方放入外部 JSON 配置：
  - 中间产物加工配方。
  - 建筑建造配方。
  - 机器人生产配方。
- 定义机器人蓝图数据结构：
  - 蓝图 ID。
  - 蓝图版本。
  - 单位属性。
  - 生产成本。
  - 规则文件路径。
  - 默认脑干开关。
- 定义队伍枚举：
  - 玩家。
  - 敌人。
  - 中立。
- 定义地图基础数据：
  - 格子尺寸。
  - 地图宽高。
  - 建筑占格尺寸。
  - 资源点类型。
  - 地图对象 ID。

### 推荐文件

```text
Scripts/core/combat_event_log.gd
Scripts/core/id_provider.gd
Scripts/data/resource_def.gd
Scripts/data/recipe_def.gd
Scripts/data/recipe_config_loader.gd
Scripts/data/unit_blueprint.gd
Scripts/data/unit_stats.gd
Scripts/data/building_def.gd
Scripts/data/map_object_def.gd
Scripts/combat/team.gd
Scripts/ui/debug_event_panel.gd
Scenes/ui/debug_event_panel.tscn
Resources/data/recipes/mvp_recipes.json
```

### 模块化要求

- 所有事件都通过统一函数记录，例如 `CombatEventLog.record(type, payload)`。
- 事件日志只关心数据，不关心 UI。
- `DebugEventPanel` 只监听或查询事件日志，不直接写入事件。
- 配方不要硬写在建筑、蓝图或产线脚本里，应从外部配置加载为 `RecipeDef`。
- 建筑和蓝图可以保存配方 ID 与成本快照，但成本来源必须是配方配置。
- 建筑占格尺寸属于建筑定义数据，不要散落在放置逻辑里。

### 美术素材需求

这个阶段只需要 MVP 美术图标：

| 素材 | 用途 | MVP 要求 |
| --- | --- | --- |
| 资源图标：铁矿 | 资源面板 | 16x16 或 32x32，灰色矿石图标 |
| 资源图标：铜矿 | 资源面板 | 16x16 或 32x32，橙色矿石图标 |
| 资源图标：铁板 | 资源面板 | 16x16 或 32x32，金属板图标 |
| 资源图标：铜线 | 资源面板 | 16x16 或 32x32，线圈图标 |
| 资源图标：建设质料 | 资源面板 | 16x16 或 32x32，方块/建材图标 |

## 十一、阶段 5：机器人组件化与默认脑干

### 目标

让机器人作为可复用单位实体运行：移动、索敌、开火、受伤、死亡、寿命到期。
本阶段建议重写最早期的纯色方块机器人原型，不以旧 `robot.gd` 的运动/攻击行为作为实现基础。旧原型只作为字段和验证经验参考。

### 功能任务

- 拆分或重写机器人底层组件：
  - HP。
  - 移动。
  - 索敌。
  - 武器。
  - 寿命。
  - 队伍与目标过滤。
  - 当前动作/调试状态。
- 实现默认脑干：
  - 找最近敌人。
  - 索敌不再使用雷达碰撞箱；从当前可见地图对应的世界范围内获得敌军单位列表。
  - 敌人在开火范围外则接近。
  - 敌人在开火范围内则停止接近并开火。
  - 默认脑干应该尽量把自己维持在射程边缘，而不是一边开火一边继续贴近敌人。
  - 推荐加入简单的期望距离参数，例如 `preferred_range_ratio = 0.85`，让机器人在射程内过近时后退或侧移，在过远时接近。
  - 移动意图和开火意图要分开，开火不应继承上一帧的接近速度。
  - 没有敌人则闲置或巡向敌巢方向。
- 实现调试用敌军单位：
  - 敌军单位与我方机器人共享同一套底层单位组件。
  - 敌军单位暂时不会攻击。
  - 敌军单位沿预设路径移动，用于测试索敌、追击、射程维持和命中。
  - 敌军单位拥有较厚血量，避免过快死亡导致调试困难。
  - 敌军单位也能被 `RobotInspector` 观测。
- 实现死亡原因：
  - 被击杀。
  - 寿命耗尽。
- 实现队伍过滤：
  - 玩家机器人不会攻击玩家单位。
  - 敌人不会攻击敌人。
- 确认机器人自由移动：
  - 机器人不按格移动。
  - 机器人可以穿过空格中心、格线和建筑周围。

  - MVP 阶段优先复用左侧 `ObjectInspector`，点击机器人或调试敌军时显示专属状态；暂不强制新建独立弹窗。
  - HP。

### 模块化要求

- `AIController` 发出意图：移动到哪里、攻击谁。
- `MovementComponent` 负责实际移动。
- `WeaponComponent` 负责冷却和开火。
- `HealthComponent` 负责死亡信号。
- `LifespanComponent` 负责寿命倒计时和寿命耗尽信号，不由 `RobotUnit` 直接维护 `Timer`。
- `DefaultBrain` 和 `PathFollowBrain` 作为独立决策节点挂在单位场景下；`RobotUnit` 只连接组件、公开状态并同步视觉。
- 玩家机器人和调试敌军必须复用同一套基础组件，不要维护两套生命/移动/索敌/受伤代码。
- 默认脑干、路径移动脑干、后续规则脑干应该是可替换决策层，不应该侵入底层单位组件。
- 索敌组件只查询当前可见地图内的 `combat_unit` 列表并按队伍过滤，不依赖 `Area2D` 雷达碰撞箱。
- 不要让 AI 直接扣敌人 HP。
- `RobotInspector` 读取单位公开状态，不直接调用 AI 决策函数；被检查对象可以是我方机器人，也可以是调试敌军。
- `RobotInspector` 在 MVP 阶段可以只是 `ObjectInspector` 的机器人/单位分支，后续再拆成独立 `Scenes/ui/robot_inspector.tscn`。
- 移动组件只使用世界坐标，不依赖网格坐标。

### 验收标准

- 机器人会移动、索敌、开火。
- 索敌结果来自当前可见地图内的敌军列表，而不是雷达碰撞箱进入/离开事件。
- 默认脑干不会在开火时继续沿上一帧接近速度贴近敌人；进入射程后应停止接近或维持在射程边缘。
- 子弹能造成伤害。
- 机器人死亡后移除或回收到对象池。
- 寿命到期会触发 `robot_lost`。
- 默认脑干在没有自定义规则时能独立战斗。
- 调试敌军能沿预设路径移动，不攻击，但能被我方机器人索敌和攻击。

- 单位头顶能显示当前动作短文本，便于开发期观察。
- 机器人不会被网格系统强制吸附。

## 十二、阶段 6：自定义规则与集结行为

### 目标

让玩家第一次通过规则改变战斗结果。

### 功能任务

  - 未到达锻造厂集结点。

- 扩展动作：
  - 前往锻造厂集结点。

  - 第一条命中后截断。
  - 没有规则命中时执行默认脑干。
- 增加规则触发气泡：
  - `集结`
  - `攻击`
  - `默认脑干`
  - 同一机器人同一规则设置 3-5 秒显示冷却，避免刷屏。
- 增加规则调试日志：
  - 规则命中。
  - 规则失败。
  - 被跳过原因。
- 在 `RobotInspector` 中显示：
  - 当前命中规则。
  - 是否回落到默认脑干。
  - 规则失败原因。

### 推荐文件

```text

Scripts/ai/ai_controller.gd
Scripts/ai/default_brain.gd
Scripts/ui/rule_bubble.gd
Scenes/ui/rule_bubble.tscn
```

### 模块化要求

- 条件判断函数不要直接操作单位状态，只返回布尔值。
- 动作执行函数可以调用单位接口，例如 `move_to(position)` 或 `attack(target)`。
- 规则系统不应该知道锻造厂生产逻辑，只读取机器人身上的 `rally_point_position`。

| 规则面板中文标签 | UI | 可沿用现有 UI |
| 规则失败提示 | UI | 可用右侧事件列表文字显示 |

### 验收标准

- 机器人能读取包含集结规则的 JSON。
- 集结规则触发时机器人读取阶段 4 注入的 `has_rally_point` / `rally_point_position`，并前往该集结点。
- 当规则不再触发时，默认脑干接管并攻击敌人。
- `rule_triggered` 事件能统计集结规则触发次数。
- 头顶气泡能让玩家看见“集结”和“默认脑干接管”的切换。
- 规则从未触发或触发失败时，调试面板能显示原因。

| `default_brain_smoke.gd` | 机器人无规则时能寻敌开火 |
| `rally_rule_smoke.gd` | 集结规则触发并移动到集结点 |
| `nest_battle_smoke.gd` | 集结后能摧毁敌巢 |
| `combat_report_smoke.gd` | 事件日志能统计生产、死亡、规则触发 |
| `ui_binding_smoke.gd` | 资源、锻造厂、事件面板能从系统状态刷新 |

不要等 UI 完成后才测试系统。经济、生产、AI、战斗都应该能用脚本验证。

## 十八、推荐第一周目标

如果按个人开发节奏推进，第一周建议只做到：

1. `CombatEventLog`。
2. `Inventory`。
3. `MainBase`。
4. `GridMap` 坐标转换与占格表。
5. `Miner`。
6. `Processor`。
7. 简单资源 HUD。
8. `DebugEventPanel`。
9. `BuildingStatusPanel` 最小版。
10. `economy_smoke.gd`。
11. `grid_map_smoke.gd`。
12. `ui_binding_smoke.gd`。

这一周的目标不是看到大战斗，而是让“资源系统可信”。只要资源地基干净，后面的锻造厂、蓝图、复盘都会舒服很多。

## 十九、最重要的范围控制

MVP 阶段不要做这些：

- 实体物流机器人。
- 多种武器。
- 多种敌巢。
- 科技树 UI。
- 建筑自由摆放的大量边界情况。
- 存档系统。
- 漂亮动画。
- 复杂窗口停靠系统。
- 可换肤 UI 框架。
- 大量图标精修。
- 完整敌人生态。

这些都很诱人，但都会稀释 MVP 的核心验证。第一版真正要证明的只有一句话：

**玩家能通过修改机器人逻辑规则，让同样的生产线打出明显不同的战斗结果。**
