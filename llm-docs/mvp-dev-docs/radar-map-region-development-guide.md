# Logic Line 雷达图标风地图区域开发指导

本文档用于指导后续 Codex 会话继续开发地图区域、第二资源圈、自然关隘和雷达图标风地貌表现。

目标不是制作写实地形，也不是把地图做成传统关卡地块；目标是在当前程序绘制网格地图上，用雷达 UI 可以读懂的符号层表达区域、地貌、威胁、资源方向和自然入口。

- [svg-art-direction-and-asset-guide.md](svg-art-direction-and-asset-guide.md)：SVG 图标风格、色彩和复杂度约束。
- [svg-art-asset-manifest.md](svg-art-asset-manifest.md)：地图底纹、信号和后续地图标记素材登记。
- [../design-docs/victory-loss-and-map-flow.md](../design-docs/victory-loss-and-map-flow.md)：单张大地图、威胁圈、软门槛和地图推进设计。

## 一、核心视觉原则

### 1. 不要画“地形”，要画“雷达对地形的解释”

项目当前地图是程序绘制网格，整体美术是 SVG 图标风和雷达 UI 风。因此地貌不应表现为写实地板、写实山体或大面积贴图，而应表现为雷达扫描出的地貌特征：

- 水：青蓝等高线、短波纹、断续岸线、水泵候选点。
- 晶体荒原：冷青/灰紫散射点、晶体簇符号、碎裂折线、断层线。
- 干扰高地：扫描线错位、断续横线、信号遮蔽、局部噪声。
- 残骸战场：破碎短线、残骸符号、危险斜纹带、断裂轮廓。
- 主脑核心外围：黑紫同心扫描圈、双环约束线、放射短线、密集高危信号。

换句话说，地图地貌是“情报图层”，不是“自然风景”。

### 2. 主地图自然，小地图清晰

主地图负责让玩家感觉自己在一张连续扩张的大地图上：

- 区域底色要低透明。
- 区域边界不要画硬边框。
- 地貌符号要稀疏，不能压过建筑、单位和资源点。
- 关隘通过水体、断层和路线负空间自然形成。

小地图负责导航和信息总结：

- 可以显示更清晰的区域轮廓。
- 可以显示区域名、威胁等级、未知信号和资源方向。
- 可以用更强的颜色边界，但仍应避免方块关卡感。

### 3. 数据可以方，视觉不能方

底层数据继续允许使用 `grid_rects`、区域块和网格索引，因为它们适合：

- 区域归属查询。
- 存档。
- 发现状态。
- 威胁等级。
- 后续路径与水体阻挡。

但玩家看到的主地图不应直接显示这些矩形。`grid_rects` 只能是逻辑层，正式视觉必须经过自然化表达。

### 4. 区域分割优先于区域装饰

后续地图打磨的重点不是“区域里面放更多花纹”，而是“区域之间用什么边界隔开，以及玩家从哪里穿过去”。

区域分割应由空间结构完成，而不是由装饰素材完成：

- 起始盆地和晶体荒原之间应有水体、晶体断层或二者组合形成的边界带。
- 起始盆地和干扰高地之间应有扫描噪声、断层线或信号遮蔽形成的边界带。
- 起始盆地和残骸战场之间应有残骸带、破碎地貌或危险斜纹形成的边界带。
- 主脑核心外围应通过黑紫约束环、污染带和高危信号表现远端边界。

这些边界应该是有厚度的地带，不是一条线，也不是一个方框。推荐宽度为 `5-15` 格，内部可以包含不可通行水体、未来不可通行断层、噪声遮罩、残骸带、敌巢信号和少量可穿越缺口。

装饰素材只能增强边界的识别度，不能替代边界本身。

## 二、推荐地图空间结构

整张地图采用：

`中心盆地 + 放射状扇区 + 局部环形威胁梯度`

第一版方位草案：

```text
              干扰高地 / 第三阶段
                    /\
                   /  \
残骸战场 ---- 起始盆地 ---- 晶体荒原 / 第二资源圈
                   \  /
                    \/
          主脑核心外围 / 终局方向
```

设计要求：

- 起始盆地位于中心或偏中心，是第一阶段基地和教学区。
- 晶体荒原位于右侧，是第二资源圈的主要展开方向。
- 干扰高地位于上方，承接后续侦察、广播和干扰反制。
- 残骸战场位于左侧，承接战场回收与装甲物流。
- 主脑核心外围位于下方或远端，是终局压力来源。
- 不做完美同心圆。
- 不做方块关卡。
- 威胁圈存在于数值、小地图和信号层，主地图地形保持不规则。

## 三、当前实现概况

当前已经有一个阶段 13 灰盒雏形：

- 地图配置：`Resources/data/maps/mvp_stage3_map.json`
  - `map_size` 为 128x128。
  - `region_block_size` 为 4。
  - 已配置 `starting_basin`、`crystal_wasteland`、`interference_highlands`、`wreckage_battlefield`、`brain_core_outer`。
  - 已配置 `region_routes`。
  - 已配置 `water_bodies` 和 `pump_candidate_cells`。
- 运行时管理：`Scripts/mvp/mvp_game_manager.gd`
  - 加载地图 JSON。
  - 初始化 `map_region_states`、`map_region_signal_cells`、`map_region_definitions`、`map_region_blocks`、`map_region_routes`、`map_water_bodies`。
  - 支持 `unknown`、`signal`、`scanned`、`visible`、`controlled`。
  - 空地检查器可查询当前格子的区域信息。
- 绘制层：`Scripts/map/grid_overlay.gd`
  - 绘制基础网格。
  - 绘制区域底色和程序花纹。
  - 绘制水体、路线、迷雾、敌巢信号。
- 检查器：`Scripts/ui/object_inspector.gd`
  - 选择空地时显示区域名、区域 ID、区域类型、威胁等级、推荐阶段和发现状态。

当前主要问题：

- 区域视觉仍以 `grid_rects` 的矩形为基础，容易显得像调试色块。
- 程序花纹在矩形内均匀重复，容易像壁纸或填充纹理，而不是雷达情报。
- 区域边界缺少“自然关隘”和地貌负空间。
- 水体仍像矩形组合，尚未形成自然水带或岸线。
- 路线只是线条提示，尚未和边界地貌形成互相支撑的入口设计。

## 四、最终表现目标

最终地图应像一张可操作的战术雷达屏：

- 背景是深色规整网格。
- 区域不是大色块，而是不同类型的扫描异常。
- 起始盆地是低噪声安全区。
- 晶体荒原向右展开，冷色、碎晶、远矿、水池共同提示第二资源圈。
- 水池切割路线，形成自然关隘，并为阶段 14 的泵和路径阻挡预留位置。
- 干扰高地在上方以扫描断裂、信号遮蔽和噪声表现。
- 残骸战场在左侧以破碎短线、残骸符号和高危斜纹表现。
- 主脑核心外围在下方以黑紫环形信号表现远端压力。
- 建筑、单位、资源点和敌巢图标始终比地貌更醒目。

玩家不读说明也应该能感到：

- 右侧是下一步扩张方向。
- 右侧不是一块方形区域，而是一片冷色晶体荒原扇区。
- 进入晶体荒原的推荐入口由水体/断层夹出。
- 还有一条更远、更危险的绕行路线。
- 下方主脑区域危险，但不是当前阶段立刻要处理的目标。

## 五、推荐绘制层级

主地图建议按以下顺序绘制：

1. 基础背景层
   - 深色底色。
   - 细网格。
   - 每 4 格一条较明显主网格线。

2. 区域暗示层
   - 只画低透明区域底色。
   - 建议 alpha 在 `0.06 - 0.16`。
   - 不画完整矩形边框。
   - 不让区域底色抢过建筑、单位和资源。

3. 区域边界特征层
   - 重点绘制边界，而不是填满内部。
   - 水体、断层、噪声带、残骸带、核心污染带应主要出现在区域边缘。
   - 边界线必须断续、破碎、局部出现，不要闭合成框。

4. 地貌符号层
   - 在区域内部稀疏放置符号。
   - 每 6-10 格出现一个符号就足够。
   - 不要匀速铺满。
   - 符号应像地图标记，不像装饰花纹。

5. 路线与关隘层
   - 主路线可以用低透明青蓝线。
   - 高风险绕行可以用低透明橙色线。
   - 路线必须穿过自然关隘，而不是随意穿过区域色块。

6. 区域边界主体层
   - 这一层优先于装饰符号。
   - 用程序绘制水道、裂谷、扫描噪声带、残骸封锁带和核心约束带。
   - 边界主体可以参与后续阻挡、寻路、建造合法性和小地图投影。

7. 局部 SVG 符号层
   - 水波、岸线短划、水泵锚点、晶体裂口、干扰 tick、残骸碎片、警示斜纹等。
   - 只用于增强识别度，不决定边界形状。
   - 沿边界和关隘布置，少量放在区域内部。

8. 发现状态和信号层
   - 未知区域遮罩。
   - 敌巢信号。
   - 高能、干扰、主脑等未知信号。

9. 游戏对象层
   - 资源点。
   - 建筑。
   - 单位。
   - 敌巢。
   - 子弹和战斗反馈。

地貌层永远不应压过游戏对象层。

## 六、数据结构建议

不要立刻删除 `grid_rects`。建议保留当前结构，并新增视觉特征层。

### 1. 继续使用 `grid_rects` 表达逻辑区域

用途：

- 区域查询。
- 区域状态。
- 推荐阶段。
- 威胁等级。
- 存档。
- 后续小地图简化。

### 2. 新增 `visual_outline_cells`

用于主地图视觉轮廓：

```json
{
  "region_id": "crystal_wasteland",
  "grid_rects": [[78, 36, 38, 50], [72, 50, 10, 20]],
  "visual_outline_cells": [
    [76, 42], [84, 34], [102, 36], [118, 48],
    [116, 84], [104, 98], [84, 92], [72, 70]
  ]
}
```

原则：

- `visual_outline_cells` 不参与区域归属判断。
- 它只负责画给玩家看的不规则扇区轮廓。
- 第一版可以只用于小地图或调试层，主地图逐步接入。

### 3. 新增 `visual_features`

用于地貌符号、边界符号和自然关隘：

```json
{
  "visual_features": [
    {
      "type": "crystal_cluster",
      "cells": [[86, 48], [94, 58], [104, 72]]
    },
    {
      "type": "fracture_line",
      "path_cells": [[78, 40], [84, 44], [92, 45]]
    },
    {
      "type": "scan_noise_band",
      "grid_rects": [[72, 50, 8, 20]]
    },
    {
      "type": "wreckage_marks",
      "cells": [[30, 56], [24, 68], [36, 74]]
    }
  ]
}
```

原则：

- `visual_features` 比区域内部花纹更重要。
- 先用少量可读 feature，避免铺满。
- feature 应服务区域边界、资源方向、关隘和威胁提示。

### 4. 新增区域连接/关隘数据

用于替代“摧毁敌巢后硬编码揭示右/下/右下区域”的做法：

```json
{
  "region_connections": [
    {
      "from_region_id": "starting_basin",
      "to_region_id": "crystal_wasteland",
      "connection_type": "main_gate",
      "gate_cells": [[72, 58], [73, 58], [74, 59]],
      "route_id": "basin_to_crystal_main",
      "revealed_by": "enemy_nest_001_destroyed"
    },
    {
      "from_region_id": "starting_basin",
      "to_region_id": "crystal_wasteland",
      "connection_type": "risk_bypass",
      "gate_cells": [[70, 78], [72, 80]],
      "route_id": "basin_to_crystal_risk_bypass"
    }
  ]
}
```

用途：

- 小地图显示主要入口。
- 敌巢摧毁后揭示下一阶段信号。
- 后续物流前哨位置建议。
- 玩家路线引导。

### 5. 新增区域边界数据

区域分割建议使用独立的 `region_boundaries`，不要只靠 `region_routes` 或 `visual_features` 表达。`region_boundaries` 描述“两个区域之间是什么地貌边界、哪些格子是边界主体、哪些位置可以穿越”。

示例：

```json
{
  "region_boundaries": [
    {
      "id": "basin_crystal_frontier",
      "between": ["starting_basin", "crystal_wasteland"],
      "boundary_type": "water_channel",
      "water_cells": [[76, 48], [77, 48], [78, 49]],
      "ridge_path_cells": [[78, 44], [80, 52], [79, 68]],
      "visual_outline_cells": [[74, 44], [80, 48], [82, 56], [78, 66]],
      "gate_cells": [[75, 59], [76, 59], [77, 60], [78, 60]],
      "bypass_cells": [[72, 78], [73, 79], [74, 80]],
      "route_ids": ["basin_to_crystal_main", "basin_to_crystal_risk_bypass"]
    }
  ]
}
```

字段含义：

- `between`：边界分隔的两个区域。
- `boundary_type`：边界类型，例如 `water_channel`、`crystal_ridge`、`signal_noise`、`wreckage_belt`、`core_containment`。
- `water_cells` / `blocking_cells`：边界主体格子，阶段 14 起可以参与不可通行、泵建造或寻路。
- `ridge_path_cells`：断层、晶体脊、裂谷或其它线性边界。
- `visual_outline_cells`：给主地图或小地图绘制的自然轮廓。
- `gate_cells`：主通道，边界在这里断开，形成自然关隘。
- `bypass_cells`：高风险绕行通道，不硬锁，但距离更远或更危险。
- `route_ids`：关联已有 `region_routes`。

第一优先级是制作 `starting_basin` 到 `crystal_wasteland` 的 `water_channel + crystal_ridge` 边界。这个边界做好后，地图会比继续增加区域内部花纹更快摆脱“颜色矩形拼接感”。

## 七、各区域的雷达地貌语言

### 起始盆地

目的：低压力、低噪声、适合建厂。

表现：

- 底色：暗灰蓝，低透明。
- 网格：略微更清晰。
- 符号：少量圆角扫描框、基础矿点图标。
- 边界：轻微盆地弧线或安全扫描圈。

避免：

- 不要铺满纹理。
- 不要使用高饱和颜色。
- 不要出现太多威胁信号。

### 晶体荒原

目的：第二资源圈，远矿、水池、前线补给和水泵。

表现：

- 底色：冷青、灰紫，低透明。
- 边界：碎裂折线、晶体脊、冷色断层。
- 内部符号：稀疏三角晶簇、短斜线、散射点。
- 水体：青蓝波纹、断续岸线、泵候选点。
- 关隘：两片水体或晶体脊夹出的窄通道。

避免：

- 不要把晶体符号均匀铺满。
- 不要把整个区域画成青色矩形。
- 不要让水体像蓝色矩形块。

### 干扰高地

目的：后续侦察、广播、干扰塔和情报层玩法。

表现：

- 底色：暗黄绿或灰绿，低透明。
- 边界：扫描线错位、信号断层、局部遮蔽。
- 内部符号：断续横线、短波纹、异常扫描点。
- 信号：干扰源附近可以有圆形噪声环。

避免：

- 不要画写实山体。
- 不要用大面积绿色斜线填充。

### 残骸战场

目的：战场回收、高价值掉落和装甲物流。

表现：

- 底色：暗橙、锈色、低透明。
- 边界：破碎短线、废墟轮廓、危险斜纹带。
- 内部符号：残骸叉线、破损矩形、小型碎片图标。

避免：

- 不要把残骸画成复杂插画。
- 不要用密集垃圾纹理铺满整个区域。

### 主脑核心外围

目的：终局远端压力和最终攻坚方向。

表现：

- 底色：黑紫，极低透明。
- 边界：双环约束线、同心扫描圈、放射短线。
- 内部符号：核心节点、密集高危信号、脉冲环。

避免：

- 不要全屏紫色压过地图。
- 不要过早表现过多终局细节。

## 八、水体与关隘表现

水体是阶段 13 和阶段 14 的关键边界语言。

### 水体显示方式

即使底层仍用矩形，也不要直接画矩形水块。显示时应转译为：

- 多段横向波纹线。
- 断续岸线。
- 边缘小点或短弧。
- 泵候选点圆环。
- 少量低透明青蓝填充。

### 水体玩法意义

阶段 13：

- 水体是区域边界和未来阻挡灰盒。
- 水泵候选点只是视觉/数据预留。
- 不要求机器人真的绕开。

阶段 14：

- 水池格对普通地面机器人不可通行。
- 水池边缘格可以建泵。
- 水体参与路径规划和建筑合法性。

### 关隘设计

关隘不应是门，也不应是锁。

推荐做法：

- 两片水体之间留出一条 3-6 格宽的陆桥。
- 陆桥旁有敌巢信号或高威胁提示。
- 主路线穿过陆桥。
- 旁边留一条绕行路线，但距离更长、信号更危险。

玩家应该感觉：

> “我可以从这里进入晶体荒原；也可以绕路，但会付出代价。”

### 边界本体与美术素材的分工

区域边界建议“程序绘制为主，美术素材为辅”。

程序绘制负责边界本体：

- 水道、水池和水体轮廓。
- 裂谷、断层和晶体脊主线。
- 信号噪声带范围。
- 残骸封锁带范围。
- 主通道与绕行通道的负空间。
- 可通行、不可通行和未来寻路代价。
- 小地图区域边界和关隘投影。

SVG 美术素材负责局部识别：

- 水波短线。
- 岸线短划。
- 水泵锚点。
- 晶体裂口小符号。
- 断层节点。
- 干扰噪声 tick。
- 残骸碎片。
- 警示斜纹。
- 核心环形节点。

推荐流程：

```text
边界数据 -> 程序画边界形状 -> 沿边界撒少量 SVG 符号 -> 留出通道
```

不要把一整条边界做成一张大素材，也不要让小 SVG 素材决定边界形状。边界是地图结构和玩法逻辑，素材只是雷达识别符号。

### 跨越边界的通道表现

跨越边界的通道不要画成门、按钮或高亮框。它应该是边界里的负空间：

- 水道、裂谷或噪声带在这里断开。
- 两侧边界更清楚，中间保持干净可走。
- 主路线用低透明断续扫描线轻轻穿过。
- 通道两端可以使用小型扫描括号、路径节点或补给点候选位置。

主通道建议：

- 宽度约 `3-6` 格。
- 视觉更稳定，颜色偏青蓝。
- 周围可以出现水体岸线、晶体脊或前线补给点候选位置。
- 不要画得比资源点、建筑和敌巢更显眼。

高风险绕行建议：

- 更远、更窄或更靠近高威胁信号。
- 路线颜色可偏橙。
- 周围出现更多噪声、断层、残骸或敌巢信号。
- 允许玩家提前进入，但让玩家直觉上知道代价更高。

起始盆地到晶体荒原的推荐结构：

```text
起始盆地 -> 水池带 / 晶体断层 -> 陆桥关隘 -> 晶体荒原
```

具体表现：

- 在起始盆地右侧做一条不规则南北向水池带。
- 水池带不要连续封死，中间留一个主陆桥。
- 下方再留一个更绕远的缺口作为高风险绕行。
- 水池右侧出现晶体断层线。
- 晶体荒原的冷色信号从断层右侧逐渐增强。
- 主路线穿过主陆桥，而不是直接画在空地上。

## 九、代码改造建议

### 1. 拆分当前 `_draw_region_visuals`

当前 `Scripts/map/grid_overlay.gd` 中 `_draw_region_visuals()` 直接对 `grid_rects` 画矩形。建议改为：

```text
_draw_region_base_tints()
_draw_region_outlines()
_draw_region_features()
```

第一步只画极低透明底色。  
第二步画不规则轮廓或断续边界。  
第三步画稀疏地貌符号。

### 2. 保留矩形调试开关

建议增加导出变量：

```gdscript
@export var show_region_debug_rects: bool = false
@export var show_region_visual_features: bool = true
```

用途：

- 开发时可以打开矩形看逻辑区域。
- 正式游玩默认关闭矩形。
- 避免为了调试方便牺牲成品表现。

### 3. 新增 feature 绘制函数

建议先实现以下 feature：

- `crystal_cluster`
- `fracture_line`
- `scan_noise_band`
- `wreckage_marks`
- `core_rings`
- `water_ripple`
- `shoreline_dashes`

每个 feature 都应简单、低成本、可程序绘制。

### 4. 水体显示自然化

当前 `_draw_water_bodies()` 直接画水体矩形。建议改为：

- 矩形只作为水体 mask。
- 对每个水体绘制多段波纹线。
- 岸线只画断续边。
- 不画完整矩形边框，除非调试开关开启。
- 泵候选点继续保留圆环，但透明度降低。

### 5. 区域揭示逻辑改用连接数据

当前 `_update_region_after_nest_destroyed()` 近似写死相邻块。后续应改为：

- 查询被摧毁敌巢所在区域。
- 查询 `region_connections` 或 `reveals_on_control`。
- 将对应区域或入口附近块标为 `signal`。
- 记录 `region_controlled` 和 `region_discovered` 事件。

### 6. 新增边界绘制入口

建议在 `GridOverlay` 中新增边界绘制阶段，并让它早于装饰素材绘制：

```text
_draw_region_boundaries()
_draw_boundary_gates()
_draw_region_features()
```

绘制职责：

- `_draw_region_boundaries()`：根据 `region_boundaries` 的 `boundary_type` 绘制水道、断层、噪声带、残骸带或核心约束带。
- `_draw_boundary_gates()`：在 `gate_cells` 和 `bypass_cells` 处留出负空间，并绘制低透明扫描提示。
- `_draw_region_features()`：只补充局部 SVG 或程序符号，不再承担区域分割职责。

第一版不需要自动生成漂亮边界，允许在 JSON 中手工配置 `water_cells`、`ridge_path_cells`、`gate_cells` 和 `bypass_cells`。后续如果地图变大，再考虑用程序根据区域轮廓生成边界。

## 十、验收标准

### 视觉验收

- 玩家不读文字也能区分五个区域的大致身份。
- 晶体荒原看起来是向右展开的冷色扇区，而不是矩形色块。
- 起始盆地到晶体荒原至少有一个自然关隘。
- 起始盆地和晶体荒原之间的分割主要由水体/断层边界带完成，而不是由区域内部花纹完成。
- 主通道是边界中的负空间，不是一个被画出来的门。
- 高风险绕行路线存在，但看起来比主通道更远或更危险。
- 水体看起来像雷达识别出的水域，而不是蓝色矩形。
- 干扰高地、残骸战场和主脑核心外围即使未填玩法内容，也能在视觉上保留方向感。
- 地貌符号不会压过建筑、单位、资源点和敌巢。

### 工程验收

- `grid_rects` 仍可用于区域查询。
- 空地检查器仍能显示区域信息。
- 新增 `visual_features` 不影响现有资源点、敌巢和建筑放置。
- 地图 JSON 中区域 ID 稳定且唯一。
- feature 坐标、路线坐标、水体坐标不越界。
- Godot headless 能加载 `Scenes/mvp/mvp_test_map.tscn`。

### 设计验收

- 不使用空气墙限制玩家。
- 高阶段区域可以提前进入，但不容易稳定开发。
- Boss/关键掉落仍是科技推进门槛。
- 关隘引导路线，但不锁死路线。
- 区域分割首先回答“这里为什么自然分开、玩家从哪里穿过去”，其次才考虑区域内部装饰。
- 小地图后续可以基于同一套区域数据实现清晰导航。

## 十一、建议开发顺序

1. 增加 `visual_features` 数据结构，但先只在 JSON 中配置晶体荒原和水体附近特征。
2. 在 `GridOverlay` 中增加 feature 绘制函数。
3. 关闭正式显示中的矩形边框，只保留极低透明区域底色。
4. 重做水体显示：波纹、断续岸线、低透明填充。
5. 为晶体荒原主入口做一处水体/断层夹出的自然关隘。
6. 为高风险绕行路增加更长路径和更危险的信号表现。
7. 再补干扰高地、残骸战场和主脑核心外围的少量符号。
8. 添加地图配置校验或 smoke test。
9. 再考虑真正小地图控件。

第一轮不要追求全区域精美。只要右侧晶体荒原、水体关隘和主/绕行路线自然了，当前“方块分区感”就会明显下降。

更新后的优先级应调整为：

1. 先实现 `region_boundaries`，尤其是起始盆地到晶体荒原的边界。
2. 再把边界绘制成水道/断层/噪声带。
3. 再把主通道和绕行通道做成边界里的负空间。
4. 最后才沿边界补充 SVG 小符号和区域内部少量地貌符号。

如果只做一轮，优先做“起始盆地 -> 晶体荒原”的 `water_channel + crystal_ridge` 边界，不要继续优先增加区域内部装饰花纹。

## 十二、不要做的事

- 不要把当前矩形花纹密度继续提高。
- 不要用装饰素材承担区域分割。
- 不要用大面积高透明色块区分区域。
- 不要把区域边界画成完整闭合框。
- 不要把跨越边界的通道画成门、按钮或高亮框。
- 不要把地貌画成写实贴图。
- 不要在主地图上显示大量文字标签。
- 不要让路线线条比资源、建筑和单位更醒目。
- 不要为了阶段 13 做完整寻路、水泵生产或复杂小地图。
- 不要把科技未解锁做成不能进入区域的硬墙。
