# Logic Line SVG 图标美术指导

本文档用于指导后续 AI 生成和人工整理游戏美术素材。

目标不是制作复杂概念插画，而是延续当前 `Resources/art/` 已经使用的 SVG 图标语言，并为中后期素材预留足够的视觉表达空间。

需要批量规划或生成素材时，使用 [svg-art-asset-manifest.md](svg-art-asset-manifest.md) 中的稳定 ID、路径、优先级和状态表。

---

## 一、现有素材基线

当前正式素材位于 `Resources/art/`：

| 目录 | 内容 | 当前示例 |
| --- | --- | --- |
| `resources/` | 原料与中间产物 | 铁矿、铜矿、铁板、铜线、建设质料 |
| `buildings/` | 地图建筑 | 主基地、采矿机、基础加工厂、机器人锻造厂 |
| `chassis/` | 机器人底盘部件 | 轻型底盘 |
| `modules/` | 武器与功能模块 | 基础步枪 |
| `blueprints/` | 组合后的机器人预览 | 基础步枪机器人 |
| `enemies/` | 正式敌方单位与巢穴 | 拾荒猎犬、敌巢 |
| `units/` | 通用单位或开发期单位 | 调试敌军 |

现有素材已经形成一套适合游戏内缩放的轻量语法：

- 使用纯 SVG。
- 以 `32x32`、`48x48`、`64x64` 和 `96x96` 为主要尺寸。
- 使用扁平色块和简单描边。
- 主要由 `rect`、`circle` 和 `path` 组成。
- 常规图标通常只包含 3-8 个可见图形。
- 不使用写实纹理、复杂阴影、位图嵌入或插画式背景。
- 通过轮廓、颜色和少量局部符号表达用途。

后续新素材必须优先与现有图标并排可读，而不是单独看起来华丽。

## 二、核心视觉原则

### 1. 图标优先

所有素材首先要满足：

- 缩小后仍能识别。
- 放进地图网格后不会与背景混在一起。
- 同类单位可以快速区分。
- 不依赖文字解释。

如果一个设计只有在放大到 `256x256` 后才看得懂，它不适合当前项目。

### 2. 用几何组合表达升级

后期视觉升级主要依靠：

- 轮廓变化。
- 图形数量适度增加。
- 模块组合更丰富。
- 专属符号逐步出现。
- 图标占用面积和结构层级增加。

不要依赖：

- 写实材质。
- 渐变高光。
- 复杂发光。
- 烟雾、粒子和大面积装饰。
- 难以稳定生成的机械细节。

### 3. 视觉信息密度逐阶段增加

前期图标应主动克制。后期单位可以增加层次，但仍然保持 SVG 图标感。

| 玩家阶段 | 常规单位建议图形数 | 建筑建议图形数 | Boss 或关键目标建议图形数 |
| --- | --- | --- | --- |
| 阶段 1：起始盆地 | 4-8 | 4-7 | 8-14 |
| 阶段 2：晶体荒原 | 6-10 | 5-9 | 10-16 |
| 阶段 3：干扰高地 | 8-13 | 7-12 | 12-20 |
| 阶段 4：主脑核心区 | 10-16 | 9-15 | 16-26 |

这里的图形数是控制复杂度的参考，不是必须精确满足的硬限制。

### 4. 后期预留的是符号组合，不是基础形状

圆形、方形、线条和简单环形可以在前期使用。需要为后期保留的是更有识别性的组合：

- 多层同心核心。
- 多臂放射结构。
- 天线阵列与信号波纹。
- 导弹舱阵列。
- 牵引钩与回收机械臂。
- 大面积护盾弧线。
- 多足堡垒轮廓。
- 终局核心的多层嵌套结构。

前期素材可以使用一个圆形核心，但不要同时叠加天线、护盾弧、导弹阵列和多层外环。

## 三、基础尺寸与背景规则

| 素材类型 | 推荐尺寸 | 背景规则 | 用途 |
| --- | --- | --- | --- |
| 资源与中间产物 | `32x32` | 使用深色圆角底板 | HUD、配方、库存 |
| 底盘与模块 | `32x32` | 使用深色圆角底板 | 蓝图编辑器 |
| 蓝图预览 | `32x32` 或 `48x48` | 优先透明背景 | 蓝图库、地图单位 |
| 普通地图单位 | `32x32` 或 `48x48` | 透明背景 | 战斗地图 |
| 普通建筑 | `48x48` | 使用深色圆角底板 | 建造栏、地图建筑 |
| 主基地 | `64x64` | 使用深色圆角底板 | 地图核心建筑 |
| 敌巢与关键目标 | `64x64` 或 `96x96` | 允许底板与边框 | 地图阶段目标 |
| UI 状态图标 | `16x16`、`20x20` 或 `24x24` | 透明背景 | 气泡、按钮、统计面板 |

背景规则需要按使用场景区分：

- 地图上需要自由移动的单位优先使用透明背景。
- 库存、部件、配方和建造按钮中的图标使用深色圆角底板。
- 同一素材如果同时用于地图和 UI，优先制作地图透明版，再由 UI 容器提供底板。

## 四、基础色板

新素材可以增加少量颜色，但应优先复用现有色板。

### 1. 通用底板与机械中性色

| 用途 | 推荐颜色 |
| --- | --- |
| 资源底板 | `#2b2f33` |
| 建筑底板 | `#24292e` |
| 蓝图部件底板 | `#1f2d3a` |
| 冷灰机械暗部 | `#56616d`、`#6f7f91`、`#7f8fa0` |
| 冷灰机械亮部 | `#bfcbd8`、`#c8d4e0`、`#dbeeff` |

### 2. 玩家阵营

| 用途 | 推荐颜色 |
| --- | --- |
| 玩家主色 | `#4e9bea` |
| 玩家辅助蓝 | `#5fa8d3`、`#7ec7ff` |
| 玩家高亮 | `#9bd3ff`、`#dbeeff` |

### 3. 敌方阵营

| 用途 | 推荐颜色 |
| --- | --- |
| 敌方暗部 | `#4b2528`、`#7f3432` |
| 敌方主色 | `#b84d3f`、`#ba4d5c` |
| 敌方高亮 | `#e26a43`、`#ff8b55`、`#ffd08a` |

### 4. 资源类别

| 类别 | 推荐颜色 |
| --- | --- |
| 铁与普通金属 | `#8f969d`、`#c0c6cc` |
| 铜与导电材料 | `#b86f37`、`#e59a52`、`#f0a65b` |
| 建设质料 | `#8abf6a`、`#9ed17f`、`#b9d98f` |
| 晶体与光学材料 | 冷青或淡青紫，避免与玩家蓝完全相同 |
| 能量与高阶电路 | 黄绿色或青绿色，仅作为局部强调色 |
| 终局材料 | 高对比淡紫或白色，仅在终局使用 |

### 5. 色彩使用规则

- 一个常规图标优先使用 2-4 个主体颜色。
- 同一图标不要同时使用过多高饱和颜色。
- 玩家阵营保持偏蓝；敌方保持偏红橙。
- 资源颜色用于表达类别，不要随意借给无关单位。
- 终局强调色不要提前大量出现。

## 五、SVG 技术约束

### 1. 允许使用

- `svg`
- `rect`
- `circle`
- `ellipse`
- `path`
- `line`
- `polyline`
- `polygon`
- `fill`
- `stroke`
- `stroke-width`
- `stroke-linecap`
- `stroke-linejoin`
- `opacity`

### 2. 默认不要使用

- 嵌入式位图。
- `<text>`。
- CSS 样式表。
- 复杂 `<defs>`。
- 渐变。
- 滤镜。
- 模糊。
- 阴影。
- 蒙版。
- 裁剪路径。
- 动画。
- 大量难以手工维护的小数坐标。

这些能力并非 SVG 不支持，而是会增加 AI 输出不稳定性、Godot 导入风险和后续人工维护成本。

### 3. 文件格式

每个文件保持：

```xml
<svg xmlns="http://www.w3.org/2000/svg" width="32" height="32" viewBox="0 0 32 32">
  ...
</svg>
```

要求：

- 文件只包含一个独立图标。
- `width`、`height` 与 `viewBox` 一致。
- 使用 UTF-8。
- 文件名使用小写英文和下划线。
- 保持路径稳定，替换现有素材时不要随意改名。

## 六、分类生成规则

### 1. 资源与中间产物

资源图标使用：

- `32x32`。
- 深色圆角底板。
- 一个清晰主体。
- 最多一个辅助高光或纹理方向。

设计方式：

| 类型 | 推荐表达 |
| --- | --- |
| 原矿 | 不规则多边形矿块 |
| 金属板 | 圆角矩形与少量横线 |
| 线缆 | 一条弯曲路径与端点 |
| 齿轮 | 简化齿轮或中心圆与外侧短齿 |
| 电池 | 矩形电芯与端子 |
| 液体 | 液滴、容器或波纹 |
| 晶体 | 2-4 个棱角分明的多边形 |
| 芯片 | 方形芯片与少量引脚 |
| 反物质 | 多层核心与少量外环，只在终局使用 |

不要为普通资源绘制复杂机械场景。

### 2. 建筑

建筑图标使用：

- 常规建筑 `48x48`。
- 主基地 `64x64`。
- 深色圆角底板。
- 正视图标式轮廓，不追求严格透视。
- 一个主轮廓和 1-3 个用途符号。

例子：

- 采矿机：机械箱体、提取架、轮子。
- 加工厂：箱体、处理窗口、状态灯。
- 锻造厂：箱体、顶部装配结构、蓝色工作区。
- 前线补给点：仓储箱体、方向箭头、补给标记。
- 研究终端：控制台、芯片符号、少量节点连线。
- 远程投送塔：塔体、上行箭头或投送轨道。

建筑升级主要增加外部附件和用途符号，不要依赖写实工业细节。

### 3. 底盘

底盘图标使用：

- `32x32`。
- 深色圆角底板。
- 只表达移动方式和承载等级。

| 底盘 | 推荐表达 |
| --- | --- |
| 轻型轮式底盘 | 单一圆形机体、四方向短连接、浅蓝核心 |
| 微型搬运底盘 | 矮宽机体、小型货架或背负接口 |
| 标准货运底盘 | 双货舱、宽轮组、矩形承载区 |
| 重型履带底盘 | 宽底座、左右平行履带、较厚装甲块 |
| 装甲回收底盘 | 厚底座、前置牵引接口、侧面防护块 |
| 反重力悬浮底盘 | 中心机体、少量分离式稳定片或外侧短弧 |
| 堡垒级多足底盘 | 大型中心体、多个粗短支撑足，仅在终局使用 |

不要让轻型底盘提前拥有重型履带、悬浮稳定片和多足结构。

### 4. 武器与功能模块

模块图标使用：

- `32x32`。
- 深色圆角底板。
- 只表达一种功能。
- 主体形状占图标面积的 45%-70%。

| 模块 | 推荐表达 |
| --- | --- |
| 基础步枪 | 长矩形枪管、短枪身 |
| 链锯 | 锯齿边缘、短握柄 |
| 背包模块 | 箱体、绑带或货格 |
| 护盾模块 | 半圆弧或单层盾面 |
| 重机枪 | 更粗枪管、双矩形弹舱 |
| 热熔激光 | 枪体、单一透镜圆、细直射线提示 |
| 冷凝凝胶喷射器 | 喷嘴、液滴或小罐体 |
| 残骸牵引器 | 钩形路径、绞盘圆 |
| 回收扫描器 | 扫描扇形或单层雷达线 |
| 目标锁定器 | 十字准星与小型传感器 |
| 导弹发射器 | 2-4 个平行矩形发射舱 |
| EMP 发生器 | 中心圆与短放射线 |
| 终局奇点模块 | 多层核心与外环，只在终局使用 |

不要在一个模块图标中混入多个独立能力。

### 5. 蓝图与地图单位

蓝图预览是“底盘 + 核心模块”的组合，不应只是把部件图标完整堆叠。

要求：

- 常规蓝图优先 `32x32` 或 `48x48`。
- 地图单位使用透明背景。
- 保留一个稳定的底盘核心。
- 只附加 1-3 个最能体现用途的模块轮廓。
- 玩家单位优先使用蓝色，敌方单位优先使用红橙色。

蓝图升级要能从轮廓上读出来：

- 轻型步枪兵：圆形轻底盘 + 单枪管。
- 集结步枪兵：仍然保持轻型轮廓，差异主要放在 UI 蓝图标记，不要伪造硬件差异。
- 战区盾卫：宽重型底盘 + 护盾弧 + 较厚武器。
- 战地搬运车：宽货运底盘 + 明显货舱。
- 装甲回收车：厚底盘 + 钩形牵引臂。
- 前线侦察兵：轻量悬浮轮廓 + 传感器或准星。
- 导弹车：宽底盘 + 平行导弹舱。
- 堡垒机甲：大型中心体 + 多足 + 多臂结构。

逻辑差异不一定需要制作新的地图单位图标。只有轮廓或模块配置真的变化时，才新增蓝图预览素材。

### 6. 敌人与敌巢

敌人保持红橙色，但不同阶段使用不同轮廓：

| 玩家阶段 | 敌人视觉语言 |
| --- | --- |
| 阶段 1：起始盆地 | 低矮、简单、有机或废料式轮廓，例如拾荒猎犬 |
| 阶段 2：晶体荒原 | 厚重装甲块、晶体尖角、飞行小体型 |
| 阶段 3：干扰高地 | 天线、信号线、远程炮口、干扰节点 |
| 阶段 4：主脑核心区 | 多层核心、多臂结构、大尺度不对称或放射结构 |

敌巢与 Boss 可以比普通单位更复杂，但仍然遵守扁平 SVG 图标语言。

### 7. UI 与状态图标

UI 图标使用：

- `16x16`、`20x20` 或 `24x24`。
- 透明背景。
- 单色或双色。
- 线条尽量不细于 `1.5`。
- 每个图标只表达一个动作或状态。

优先准备：

- 蓝图入口。
- 统计入口。
- 集结。
- 等待。
- 默认脑干接管。
- 撤退。
- 运输。
- 回收。
- 护盾。
- 过热。
- 弹药不足。
- 广播。
- 接收。
- 目标锁定。
- 科技解锁。

## 七、四阶段视觉预算

为了避免前期素材侵占后期表达空间，新增素材时遵守以下预算。

| 视觉元素 | 阶段 1 | 阶段 2 | 阶段 3 | 阶段 4 |
| --- | --- | --- | --- | --- |
| 简单圆形核心 | 可用 | 可用 | 可用 | 可用 |
| 单层外环 | 少量可用 | 可用 | 可用 | 可用 |
| 重型履带或宽装甲块 | 保留 | 主要使用 | 可用 | 可用 |
| 晶体尖角 | 保留 | 主要使用 | 少量可用 | 可用 |
| 护盾弧线 | 保留 | 开始使用 | 可用 | 可用 |
| 货舱与补给箭头 | 保留 | 主要使用 | 可用 | 可用 |
| 天线阵列与信号线 | 保留 | 少量预告 | 主要使用 | 可用 |
| 准星与目标标记 | 保留 | 少量使用 | 主要使用 | 可用 |
| 导弹舱阵列 | 保留 | 保留 | 主要使用 | 可用 |
| 回收钩与机械臂 | 保留 | 保留 | 主要使用 | 可用 |
| 多层同心核心 | 保留 | 保留 | 少量使用 | 主要使用 |
| 多臂放射结构 | 保留 | 保留 | 少量使用 | 主要使用 |
| 多足堡垒轮廓 | 保留 | 保留 | 保留 | 主要使用 |
| 终局淡紫或白色强调色 | 保留 | 保留 | 保留 | 主要使用 |

“保留”表示不要用于常规单位，不代表完全禁止在极少数剧情提示中出现。

## 十、生成与验收流程

### 1. 先做全阶段缩略图占位

可以一次性为所有阶段生成低成本 SVG 草案，用于检查：

- 轮廓是否重复。
- 前期是否过于复杂。
- 后期是否仍有明显升级空间。
- 同一类别能否形成家族感。

这些草案不必立即接入游戏。

### 2. 正式素材按开发阶段接入

每次只清理、验收和接入当前开发阶段真正需要的素材。

### 3. 每张图标至少通过以下检查

- XML 可以解析。
- Godot 可以导入。
- 在实际地图缩放比例下可辨认。
- 与同阶段素材并排时不会混淆。
- 与前后阶段素材并排时层级合理。
- 没有提前消耗后续阶段的专属符号。
- 文件路径符合目录规则。
- 替换已有素材时保持文件名稳定。

### 4. 建议维护一张对照表

每次增加新素材后，将以下内容放在同一张人工检查表中：

- 全部玩家底盘。
- 全部主要战斗蓝图。
- 全部物流蓝图。
- 全部敌人。
- 全部建筑。
- 每阶段资源。

SVG 风格的优势是可以稳定维护和派生。只要持续检查同屏辨识度，就不需要依靠一次性生成全部正式素材来保证一致性。
